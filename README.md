# Прогнозирование прогрессии заболевания после радиохирургии (Gamma Knife)

Бинарная классификация интракраниальной прогрессии у пациентов после
радиохирургического лечения метастазов в головной мозг, на реальных
клинических данных ([Kaggle: Gamma Knife](https://www.kaggle.com/competitions/gamma-knife-4/overview)).

## TL;DR

Проект не про то, как получить высокую метрику — а про то, как **не обмануть
себя** на грязных медицинских данных.

Главный результат: обнаружены **4 источника утечки таргета (target leakage)**.
Самый крупный из них раздувал balanced accuracy с **0.59 до 0.84** на публичном
лидерборде Kaggle — то есть модель без аудита утечки выглядела бы «отличной»,
будучи бесполезной в момент реального клинического прогноза.

После устранения утечек честная модель достигает **balanced accuracy 0.59**
(против baseline 0.50). Анализ показывает, что слабый сигнал — не следствие
плохой модели, а свойство данных: часть меток искажена **смещением наблюдения**
(конкурирующие риски), а бинарная постановка теряет информацию о времени
до события.

## Ключевые находки

**1. `Число РХ процедур` — утечка, а не предиктор.**
Повторную радиохирургию назначают *вследствие* прогрессии, поэтому признак
является следствием исхода. На момент прогноза (сразу после первой процедуры)
его значение ещё не существует. Вклад в метрику: **+0.18 BA** на train,
**+0.25 BA** на лидерборде Kaggle (0.59 → 0.84).

**2. Парадокс индекса Карновского.**
Более высокий индекс общего состояния связан с *более* частой прогрессией —
направление, обратное клиническому ожиданию. Наиболее вероятное объяснение:
тяжёлые пациенты выбывают из наблюдения (умирают от системного
прогрессирования) до того, как внутричерепная прогрессия успевает проявиться.
Эффект подтверждён тремя независимыми методами: EDA, коэффициенты модели,
permutation importance.

**3. Потолок качества задан данными, а не моделью.**
Ни выбор более сложной модели (градиентный бустинг не превосходит логистическую
регрессию, Wilcoxon p=0.68), ни подбор гиперпараметров не выводят метрику за
пределы простого baseline. Это подтверждено статистически.

## Что демонстрирует проект

- **Аудит утечек таргета** с количественной оценкой вклада каждого источника
- **Очистка реальных грязных данных**: Excel-артефакты (`#REF!`), несогласованный
  регистр, десятичная запятая, нарушения хронологии, смешанные типы в колонках
- **Feature engineering** из временных интервалов между клиническими событиями
- **Разрезание признаков по времени прогноза** для устранения утечки без потери
  информации (события до/после радиохирургии)
- **Корректная валидация**: RepeatedStratifiedKFold, подбор порога на out-of-fold,
  сравнение моделей через тест Уилкоксона
- **Интерпретация модели**: odds ratios + permutation importance, проверка
  устойчивости коэффициентов по фолдам
- **Честный анализ ограничений**: смещение наблюдения, цензурирование,
  малый объём выборки

## Данные

- 872 пациента (680 после удаления строк с неизвестным исходом)
- 12 клинических признаков после очистки и feature engineering
- Дисбаланс классов 65/35 (прогрессия / нет)
- Метрика соревнования: balanced accuracy

## Структура ноутбука

1. **Первичный аудит и конструирование таргета** — включая аудит утечек
2. **Очистка данных** — обработка грязи с обоснованием каждого решения
3. **Исследовательский анализ** — проверка 5 клинических гипотез
4. **Построение моделей** — baseline → логрег → градиентный бустинг
5. **Подбор гиперпараметров** — и демонстрация его пределов
6. **Интерпретация модели** — факторы, влияющие на прогрессию
7. **Итоговые выводы и ограничения**
8. **Предсказание на тестовой выборке**

## О двух сабмишенах

Репозиторий содержит два файла предсказаний, и это осознанное решение:

- `submission_honest.csv` — **основная модель проекта**, без признаков-утечек.
  Клинически валидна: использует только информацию, доступную на момент прогноза.
- `submission_with_leak.csv` — версия с признаком `Число РХ процедур` для
  формального лидерборда. Тестовая выборка соревнования содержит этот признак,
  заполненный пост-фактум, поэтому метрика (0.84) отражает **утечку в тестовых
  данных, а не предсказательную способность модели**.

Разрыв 0.59 vs 0.84 — количественная иллюстрация того, во что обходится
незамеченная утечка.
